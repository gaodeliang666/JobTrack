# JobTrack Codex 开发规则

## 1. 适用范围与项目定位

本文件是 JobTrack 仓库长期适用的 Codex 开发规则。进入仓库后，应先阅读本文件、`README.md`、`CHANGELOG.md` 和 `docs/version-plan.md`。

JobTrack 是面向个人求职者的求职过程管理项目，目标是帮助用户记录公司、岗位、投递、面试和沟通信息，并在后续版本提供个人维度的统计能力。它不是招聘网站、企业 ATS、人力资源系统或多组织协作平台，不承担职位抓取、简历投递、招聘方管理和商业撮合职责。

当前 v0.1.0 只建设项目说明、范围、版本和开发流程文档。仓库尚无 Spring Boot 工程骨架，暂时不可启动。不得把计划功能描述成已经实现。

## 2. 技术栈边界

只有在对应版本计划和具体任务均经人工确认后，才允许使用以下技术：

- Java 21；
- Spring Boot 3.x；
- Maven；
- MyBatis；
- MySQL；
- Bean Validation；
- OpenAPI；
- JUnit 5、Mockito、MockMvc。

未经人工明确确认，禁止主动引入：

- 微服务、服务注册与配置中心、API 网关；
- 消息队列、Elasticsearch、分布式事务、分布式任务；
- Redis 或其他缓存中间件；
- Kubernetes、云厂商专用服务；
- 前端框架、移动端工程；
- ORM 替代方案、代码生成器、复杂设计模式；
- 当前任务不需要的生产依赖、插件或基础设施。

引入任何生产依赖前，计划必须说明用途、替代方案、涉及文件、维护成本和风险，并等待人工确认。不得仅为方便而扩大技术栈。

## 3. 任务与修改流程

每次修改必须遵循以下顺序：

1. 阅读项目规则和当前版本计划；
2. 执行 `git status` 并确认当前分支，不得在 `main` 上开发；
3. 检查已有修改，保留用户的无关改动；
4. 针对一个明确任务制定计划；
5. 在计划中列出需求理解、预计文件、业务规则、接口与 SQL 变化、异常、测试、风险和验收标准；
6. 等待人工明确确认计划；
7. 仅在确认的范围内修改并验证；
8. 展示状态和差异，停在未提交状态等待人工验收。

一次只处理一个明确任务。不得夹带重构、依赖升级、格式化或其他功能；不得修改无关文件。需求变化时先更新计划并重新等待确认。

## 4. 分层职责

未来创建后端代码时必须保持以下边界：

- Controller：接收请求、执行输入校验、调用 Service、转换并返回响应；不得包含业务规则、数据库访问或事务控制。
- Service：实现业务规则、数据归属校验、状态流转、事务边界和业务异常；不得把授权判断下放给客户端。
- Mapper（含 XML）：只负责持久化访问和结果映射；不得承载跨业务流程编排或绕过 Service 的权限规则。

不得让 Controller 直接调用 Mapper。接口路径、统一响应结构或分层边界发生变化时，必须在计划中明确说明并获得人工确认。

## 5. 用户数据隔离与 SQL

所有用户数据的读取、新增、更新、删除和统计都必须以服务端可信的当前用户标识为准。Company、Job、Application 必须直接校验 `userId`；Interview、Communication 必须通过所属 Application 校验到同一 `userId`。批量操作、导出、统计、文件访问和缓存键（如未来引入）同样必须包含用户边界。

禁止信任请求体传入的 `userId`，禁止仅凭业务对象 ID 访问数据，禁止为通过测试而删除用户隔离条件。更新和删除操作必须同时使用对象 ID 与 `userId`（或等价的归属链）限制影响范围，并检查实际影响行数。

所有 SQL 值必须使用 MyBatis 参数绑定。禁止拼接用户输入、排序字段、表名或查询条件；确需动态标识符时，只能使用服务端固定白名单。模糊查询也必须通过绑定参数传值。新增或修改 SQL 时必须覆盖越权、空结果、边界值和影响行数验证。

## 6. 异常与日志

- 区分参数校验错误、业务错误、资源不存在、无权访问和系统错误；响应不得泄露内部实现细节。
- 禁止吞掉异常，禁止捕获异常后返回成功，禁止用笼统异常掩盖可识别的业务错误。
- 事务失败必须保持数据一致性；不得在失败后伪造成功结果。
- 只为关键操作记录必要日志，日志应包含可追踪的业务对象 ID、脱敏后的用户标识和错误原因。
- 禁止在日志或异常中记录密码、Token、API Key、Cookie、Authorization 头、完整简历、身份证号或数据库连接凭据。

## 7. 敏感信息保护

禁止把真实密码、Token、API Key、私钥、证书、Cookie、个人隐私数据、生产数据库地址或本机绝对路径提交到仓库。敏感值只能通过环境变量或未纳入版本控制的本地配置注入。

示例配置必须使用明显的占位值并允许提交，不能复用真实信息。修改完成后必须搜索常见密钥格式、敏感字段和绝对路径；发现疑似泄露时立即停止，不得提交或推送，并报告人工处理。

## 8. 测试与人工验收

每个功能至少覆盖正常、参数非法、数据不存在、重复数据、非法状态、越权访问、边界数据和持久化结果。涉及 SQL 时必须验证 `userId` 条件；涉及事务时必须验证回滚和一致性。

按任务风险运行适用的自动化测试，并报告命令与结果。无法运行时必须说明原因、未验证范围和人工验证方法。自动化测试通过不等于版本验收通过；必须展示 `git status` 和完整差异，由人工按 `docs/acceptance-checklist.md` 验收后才能进入 Git 发布流程。

## 9. Git 与发布规则

- 禁止直接在 `main` 分支开发或提交；每个版本、功能或修复使用独立分支。
- 分支命名使用 `feature/`、`fix/`、`test/`、`refactor/`、`docs/` 或 `chore/` 加清晰主题。
- 一个 Commit 只表达一个目的；Commit Message 使用 `feat`、`fix`、`test`、`refactor`、`docs` 或 `chore` 等约定前缀。
- 未经人工明确指令，Codex 不得执行 `git add`、`git commit`、`git push`、`git merge`、`git rebase`、创建 Pull Request、创建 Tag 或 GitHub Release。
- Push 前必须人工确认 Commit 和目标远程分支；禁止强制推送和重写已共享历史。
- Merge 前必须通过自动化检查和人工验收；不得绕过评审直接合并到 `main`。
- 每个版本必须更新 `CHANGELOG.md`；只有版本内容验收并合并后，才允许按人工指令创建语义化版本 Tag。
- Tag 必须指向已验收的发布提交，名称使用 `vX.Y.Z`；Tag 和 GitHub Release 均不得宣称未完成能力。

## 10. 完成时的交付信息

完成任务后应报告：修改文件及目的、Git 状态和差异、接口和数据库变化、测试命令与结果、人工验收步骤、风险、未完成项，以及建议的 Commit Message 和版本号。除非人工明确要求执行 Git 操作，否则保持未提交状态。
