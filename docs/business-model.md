# JobTrack 业务模型

## 1. 业务价值

JobTrack 的核心价值是让个人求职者用一条清晰、可回顾的记录链管理求职机会：从关注公司和岗位，到投递、沟通、面试，再到结果复盘。当前项目不设计招聘方收费、广告、交易或商业撮合模式。

## 2. 用户旅程

计划中的典型流程如下：

1. 用户建立本人账户或身份上下文；
2. 用户记录关注的公司；
3. 用户在公司下记录岗位；
4. 用户针对岗位创建投递记录并维护状态；
5. 用户在投递下记录沟通和面试；
6. 系统基于该用户的数据生成进度和结果统计。

该流程是后续版本的业务蓝图，v0.1.0 不提供可操作页面、接口或数据库实现。

## 3. 对象关系

```text
User 1 ── N Company
User 1 ── N Job
Company 1 ── N Job
User 1 ── N Application
Job 1 ── N Application
Application 1 ── N Interview
Application 1 ── N Communication
Statistic = 当前 User 范围内数据的派生结果
```

所有业务对象最终都必须归属到唯一 User。跨用户关联、查询、统计、修改和删除均不允许。

## 4. 核心规则

### Company 与 Job

- Company 是用户维护的求职上下文，不是全平台共享的主数据。
- Job 必须关联当前用户可访问的 Company。
- 同名公司或岗位是否去重，由具体实现版本定义，不能在 v0.1.0 中假定。

### Application

- Application 表示用户对一个 Job 的一次投递或跟进机会。
- 是否允许同一 Job 多次投递、状态枚举及流转规则，由实现版本确认。
- 状态变化应保留必要时间信息，非法状态流转必须被拒绝。

### Interview 与 Communication

- 两者必须依附于当前用户拥有的 Application。
- 创建或修改前必须验证完整归属链，不能只校验 Application ID 是否存在。
- 时间、结果、内容长度和重复规则由对应版本定义并测试。

### Statistic

- Statistic 是派生结果，不是跨用户共享数据。
- 统计口径、时间范围、空数据表现和时区必须在实现版本中明确。
- 所有统计 SQL 都必须带当前 `userId` 限制或等价的安全归属链。

## 5. 数据隔离与信任边界

- 当前用户身份必须来自服务端可信认证上下文。
- 客户端传入的 `userId` 不能作为授权依据。
- 读取、写入、批量操作、导出和统计均要验证用户归属。
- 对无权访问的对象不得泄露敏感内容或其他用户信息。
- SQL 使用参数绑定；动态排序或字段仅允许服务端白名单。

## 6. v0.1.0 目标与验收标准

### 目标

定义稳定的业务对象、关系、用户旅程和安全边界，但不提前固化尚待确认的字段、状态与数据库设计。

### 验收标准

- 七类核心对象及其关系表述清楚；
- 每类数据都能追溯到唯一 User；
- 明确总体业务蓝图在 v0.1.0 尚未实现；
- 未虚构字段、接口、数据库表或商业能力；
- 与产品范围和开发规则保持一致。
