# JobTrack 开发流程

## 1. 基本原则

- 禁止直接在 `main` 分支开发。
- 一次只处理一个明确任务，不夹带重构、依赖升级或无关格式化。
- 每次先制定计划，等待人工明确确认后再修改。
- 只修改已确认范围内的文件，保留工作区中的用户改动。
- 计划能力不得描述为已经完成。

本流程与仓库根目录的 `AGENTS.md`、`README.md` 保持一致；若规则需要改变，应在同一个经确认的文档任务中同步更新。

## 2. 开始任务

1. 阅读 `AGENTS.md`、`README.md`、`CHANGELOG.md` 和 `docs/version-plan.md`。
2. 执行 `git status`、`git branch --show-current` 和必要的历史检查。
3. 确认当前不是 `main`，并识别已有修改和未跟踪文件。
4. 将需求收敛为一个可独立验收的目标。
5. 输出计划，至少说明需求理解、业务对象、预计文件、接口与 SQL 变化、业务规则、异常、测试、风险和验收标准。
6. 等待人工明确确认；确认前不修改项目文件。

如果实现中发现范围、技术方案或风险发生实质变化，应停止修改、更新计划并重新等待确认。

## 3. 实施边界

只执行计划内的最小修改。引入任何生产依赖前，说明用途、替代方案、涉及文件、维护成本和风险。未经确认，不主动引入微服务、消息队列、Elasticsearch、Redis、分布式事务、Kubernetes、前端框架或额外基础设施。

未来后端分层必须遵守：

- Controller 只处理请求、校验、Service 调用和响应转换；
- Service 处理业务规则、数据归属、状态流转、事务和业务异常；
- Mapper（含 XML）只负责持久化访问和结果映射；
- Controller 不得直接调用 Mapper，业务逻辑不得下沉到 Controller 或 Mapper。

## 4. 数据与安全检查

- 当前用户身份来自服务端可信上下文，不信任请求中的 `userId`。
- Company、Job、Application 直接校验 `userId`；Interview、Communication 通过 Application 校验完整归属链。
- 查询、新增、更新、删除、批量操作、统计、导出和文件访问均保持用户隔离。
- SQL 值全部使用参数绑定；动态标识符只允许服务端固定白名单。
- 更新和删除同时限制对象 ID 与用户归属，并检查影响行数。
- 不记录或提交密码、Token、API Key、Cookie、私钥、完整隐私数据、生产连接信息或本机绝对路径。
- 示例配置使用占位值并保持可提交，真实敏感配置只从环境变量或本地忽略文件读取。

## 5. 异常、日志与测试

区分参数错误、业务错误、不存在、无权访问和系统错误。禁止吞异常、失败后返回成功或向响应泄露内部细节。关键日志只记录必要的业务对象 ID、脱敏用户标识和错误原因，不记录敏感信息。

每个功能至少验证：正常、参数非法、数据不存在、重复、非法状态、越权、边界和持久化结果。SQL 变更必须验证 `userId` 条件，事务变更必须验证回滚。运行与风险匹配的自动化测试；不能运行时，说明原因、未验证范围和人工步骤。

## 6. 完成与人工验收

1. 检查只修改了计划内文件。
2. 搜索密码、Token、API Key、私钥、生产连接信息和本机绝对路径。
3. 检查没有新增未确认的代码、依赖、配置或基础设施。
4. 执行适用的测试和静态检查。
5. 执行并展示 `git status`、`git diff --stat`、`git diff` 和必要的提交历史。
6. 报告修改文件、接口与数据库变化、测试结果、风险、未完成项和人工验收步骤。
7. 停在未提交状态，等待人工按 `docs/acceptance-checklist.md` 验收。

自动化检查通过不等于人工验收通过。

## 7. Git 与发布

- 使用 `feature/`、`fix/`、`test/`、`refactor/`、`docs/` 或 `chore/` 分支；一个 Commit 只表达一个目的。
- Commit Message 使用 `feat`、`fix`、`test`、`refactor`、`docs` 或 `chore` 等约定前缀。
- 未经人工明确指令，不执行 `git add`、Commit、Push、Merge、Rebase、Pull Request、Tag 或 GitHub Release。
- Push 前人工确认 Commit 和远程目标；禁止强制推送及重写共享历史。
- Merge 前必须通过自动化检查和人工验收，不得绕过评审直接合并 `main`。
- 每个版本更新 `CHANGELOG.md`；只有验收并合并后，才能按人工指令创建 `vX.Y.Z` Tag 与 Release。
